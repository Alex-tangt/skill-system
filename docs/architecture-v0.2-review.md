## 已采纳的批评及解决方案

### 1. 过度抽象 ✓

你说得对——4 个 ABC 各 1 个实现是过度设计。

- **BaseTrigger**：直接砍掉。只有 ManualTrigger，Cron/Event 驱动是远期的事。触发策略的抽象现在没有任何价值。
- **BaseExtractor + BaseDedup**：保留。这两个是数据流水线的核心可变策略接口。提取器从正则换到 LLM 是明确的 upgrade 路径，去重从 SHA256 换到语义去重也同理。在切换算法时不改调用方，这是接口真正的价值。但你说得对——如果短期内只有一个实现，就不该抽——我们只保留"有明确第二个实现计划"的接口。

### 2. MCP 做进程内通信 ✓

完全同意这是一个设计错误。Internal plugin 走 `call_plugin_tool()` → JSON 序列化是多余的绕圈。修复方案：internal plugin 直接 Python 方法调用，`list_mcp_tools` / `call_tool` 只服务于 external plugin 模式。

### 3. SkillTriggerExtractor 在 v0.2 失效 ✓

确实。这是 v0.1 时代的代码没有随 v0.2 更新的 bug。SkillTriggerExtractor 匹配的是 `mcp__.*__skill_execute`，但 v0.2 不再有这个工具。需要在 trace 测试前重写匹配逻辑——改为匹配 skill scripts 路径而非 MCP 工具名。

### 4. 16 个 MCP 工具太多 ✓

采纳。而且做了更深层的划分——不是所有工具都是给 LLM 看的：

```diff
- 给 LLM: 16 个工具（含 plugin_list, pipeline_run, trace_get 等内部管理工具）
+ 给 LLM: skill_list, skill_get, skill_search（只读，内核登记处）
+ 内部: pipeline 管理、trace 查询、插件管理 → Bash 调 Python 函数
```

进一步讨论中得出结论：`skill_create` / `skill_update` / `skill_delete` 也应从内核移除——写操作属于特定领域插件（skill-creator、optimizer），内核只管登记和查询。

### 5. Hook 静默丢数据 ✓

`capture.py` 确实需要持久日志文件——当前异常只打 stderr，hook 不崩溃但静默丢数据。加本地日志文件，不在本次评审修改范围内但记录为必要改进。

### 6. 端到端优先 ✓

完全同意。我们已把"真实场景数据测试"设为当前唯一高优先级待办（#P-10）。在完整链路（Hook → History DB → Pipeline → Trace DB）没有跑通并产出有价值数据之前，架构讨论到此为止。

## 部分有保留意见的回应

### 关于 TF-IDF 检索

你的观点（skill < 100 时 LLM 直接看 L1 就够了）在实践上是对的。当前 3 个 skill 完全不需要检索。

但保留它的成本很低——代码已写、测试已过。当 skill 数量到 50+ 时，它作为 `skill_search` 的快速 pre-filter 仍有价值。不删，低优先级保持。

### 关于 skill 归属推断

你指出的问题是真实存在的——hook 不暴露 skill 上下文，归属只能推断。

但你把"归属推断"和"描述优化"串在了一起，说前者是后者的前提。我们仔细拆开后发现两者是**正交问题**：

- **归属推断**：这条工具调用属于哪个 skill？——这是 trace 提取的事，处理已发生的事件。
- **描述优化**：某个 skill 本应被触发但没被触发——这是找"未发生的事"。你不能从 hook 日志里找到不存在的事件。

描述优化需要回答的不是"这归谁"，而是"在这个对话上下文里，agent 实际手动做了哪些操作，有没有一个现有 skill 可以替代它"。这不需要 skill 归属推断，需要的是**能力覆盖分析**。

### 关于编排

你的建议（推迟到有真实场景驱动时再设计）我们采纳。这是 v0.2 文档里最早标记为"待探索"的问题，不会在 v0.2 解决。

## 深入的开放问题

讨论中我们衍生出三个更深的问题。目前没有成熟答案，但方向在成形。

### 问题 1：从"匹配触发条件"到"划分 task → 提取 trace"

我们最初的想法是：为每个 skill 定义触发条件，扫描对话，做语义匹配。这个思路有个根本性的扩展性问题：

```
复杂度 O(S × C × M)

S = skill 数量, C = 对话片段数, M = 单次语义匹配成本
100 skill × 1000 段对话 = 10 万次 embedding 比较
```

更好的方向是"先划分 task、再提取 trace、最后判断匹配"：

```
复杂度 O(C_task + K × S)

C_task = 一次线性扫描切分对话
K      = 每段对话产生的 task 数（通常远小于滑动窗口数）
S      = skill 总数
```

关键是：不追问"这段对话像不像 skill X 的触发条件"（模糊的 open-ended 匹配），而是问"这段对话里 agent 实际做了什么，有没有一个 skill 能覆盖它"（精确的 closed 匹配——答案空间是有限的）。而且 task 划分 + trace 提取本身值钱——一次处理同时服务于归属推断和描述优化。

这里你的批评是对的——我们目前把"归属推断"的难度估计太低了，但这个方向比语义匹配更可行。

### 问题 2：对话如何划分 task？

**现状：没有公认标准。** 这是整个 agent 工具链的空白地带。Claude Code、Cursor、Copilot 都没有做 task 划分——只有连续的 session。

可能的切分信号（都不够好）：

| 信号 | 可靠度 | 问题 |
|------|-------|------|
| 用户新消息 | 高 | 不全是新 task，有时是追问 |
| 工具调用语义聚类 | 中 | 同一 task 用多个工具，多个 task 可能用同一工具 |
| LLM 意图切换检测 | 中 | 额外 LLM 调用，成本高 |
| 时间间隔 | 低 | 不够精准 |

这个问题的难度可能超过"描述优化"本身。我们目前倾向于：task 划分不作为 pipeline 步骤，而是 optimizer 的第一步预处理。先积累数据，再研究划分算法。

### 问题 3：遗漏 vs 新情况

这是三个问题里最根本的。对于对话中 agent 手动执行的操作：

```
如果有一个现有 skill 能覆盖这些操作 BUT skill 没被触发
  → 遗漏 → 进入优化池（改进该 skill 的 description）

如果没有任何现有 skill 能覆盖
  → 新情况 → 新 skill 候选（这是比描述优化更有野心的方向）
```

判断"现有 skill 能否覆盖"不是语义匹配，而是**能力映射**：把 skill 的 scripts/ 和 agent 实际执行的工具调用做对比。agent 手动跑了 `Bash(pytest tests/)` → 查所有 skill 的 scripts/ 里有没有 pytest 相关脚本 → 有 run-tests skill 但没被触发 → 遗漏。

对于无法被任何 skill 覆盖的情况——"新 skill 候选生成"——这是自动从用户行为中发现 skill 需求。更难，但更有价值。

## 总结

| 评审点 | 决策 |
|--------|------|
| 过度抽象（BaseTrigger） | 采纳，砍掉 |
| MCP 内部通信 | 采纳，改直接方法调用 |
| SkillTriggerExtractor 失效 | 采纳，重写匹配逻辑 |
| 16 个工具太多 | 采纳，缩到 3 个给 LLM |
| Hook 日志 | 采纳，加持久日志 |
| 端到端优先 | 采纳，当前唯一高优 |
| TF-IDF 价值有限 | 保留但低优 |
| 归属推断 → 描述优化 | 不同意。两者正交，归属推断 ≠ 描述优化的前提 |
| 编排语法 | 采纳推迟 |
| 插件隔离 | 采纳推迟，记录为 #P-11 |

**最关键的开放挑战**：对话如何划分成 task，以及如何判断 agent 手动执行的操作能不能被现有 skill 替代。这两个问题我们目前没有答案，但认为"分 task + 能力映射"的方向比"语义匹配触发条件"更可行和更通用。
